# gx10 throughput baseline — 2026-05-16

Goal: pin down where we stand on speed across the three demo paths
(native Mat / CUDA FFI / planned SmolLM2 FFI). All numbers below from
`gx10` (NVIDIA GB10 / aarch64 / 20 cores), commit `81460c3`.

Prompts: "Hello, my name is" for GPT-2, "Once upon a time" for SmolLM2.
Each demo generates 8 new tokens.

## Results

| Demo | Model | Backend | Wall / 8 tok | Per-token | Tok/s |
|---|---|---|---:|---:|---:|
| `demos/gpt2_pleasant` | distilgpt2 (82M) | native Mat (Ruby) | 4763 ms | 595 ms | **1.68** |
| `demos/smollm2_pleasant` | SmolLM2-135M | native Mat (Ruby) | 7010 ms | 876 ms | **1.14** |
| `demos/distilgpt2_demo_kv_cuda` | gpt2-small (124M) | FFI KV-cache via ggml-CUDA | 270 ms / 29 tok | **9.3 ms** | **107.4** |

For reference, the same demos on Mac (M2 Air, all CPU):

| Demo | Mac Per-token | gx10 Per-token | Speedup |
|---|---:|---:|---:|
| `gpt2_pleasant` (native Mat) | 1768 ms | 595 ms | 3.0× |
| `smollm2_pleasant` (native Mat) | 2654 ms | 876 ms | 3.0× |

## Reading

* **Native Mat path is for clarity, not speed.** Ruby double-loop
  matmuls. ~1 tok/s at small-model scale. Educational; not useful
  for serving.
* **FFI + KV-cache + CUDA is fast.** 107 tok/s on gpt2-small is in
  the same order of magnitude as llama.cpp at this shape (typically
  150–250 tok/s on a comparable GPU). The gap is mostly: we don't
  yet fuse ops, we re-build the graph per step (`tnn_reset_for_rebuild`
  has measurable overhead at ~14k tensor slots), and we use F32 (no
  Q8_0 dequant fast path yet wired through).
* **SmolLM2 has no FFI/CUDA path yet.** This is what blocks reasonable
  perf for the llama-family. The architecture's there
  (`Toy::SmolLM2` / `Toy::GQAttention` / `Toy::SwiGLU` /
  `Toy::RoPE`); what's missing is the parallel ggml graph builder.

## Side issues

* **CPU FFI build on gx10 fails** with `undefined reference to GOMP_parallel`.
  Old `vendor/ggml/build/` was built without `-DGGML_OPENMP=OFF`. Fix:
  `rm -rf vendor/ggml/build && make setup-ggml` to pick up the
  current `setup-ggml` flags (which set `GGML_OPENMP=OFF`). Doesn't
  affect CUDA — `distilgpt2_demo_kv_cuda` builds and runs.

## Next step

Build `Toy::SmolLM2KVFFI` (and ideally `Toy::SmolLM2KVFFICuda`) —
mirror of `lib/gpt2_ffi_kv.rb` / `lib/gpt2_ffi_kv_cuda.rb` but for
the llama-family ops. New ggml-side wrappers needed:

* `tnn_silu` (one line: `ggml_silu`)
* `tnn_mul`  (one line: `ggml_mul`)
* `tnn_rope_ext` (RoPE on Q/K — already in ggml, just bind it)

Then the KV-cache layout becomes `n_kv` (not `n_heads`) wide — GQA
saves real KV-cache bytes at this point.

Plausible target: **30–60 tok/s** for SmolLM2-135M on GB10 (roughly
in the ratio between gpt2-small CUDA perf and the 135M size).
Llama.cpp territory at this shape is ~300–500 tok/s; we won't match
that without op fusion, but 50 tok/s would be very respectable for
a Ruby project.
