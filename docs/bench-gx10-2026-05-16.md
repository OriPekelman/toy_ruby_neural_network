# gx10 throughput baseline — 2026-05-16

All numbers from `gx10` (NVIDIA GB10 / aarch64 / 20 cores) and Mac M2 Air.
Prompts: "Hello, my name is" for GPT-2, "Once upon a time" for SmolLM2.

## Per-token decode throughput

| Demo | Model | Path | Per-token | Tok/s |
|---|---|---|---:|---:|
| `demos/gpt2_pleasant`        | distilgpt2 (82M) | native Mat (Mac CPU) | 1768 ms | 0.6 |
| `demos/gpt2_pleasant`        | distilgpt2 (82M) | native Mat (gx10 CPU) | 595 ms  | 1.7 |
| `demos/smollm2_pleasant`     | SmolLM2-135M     | native Mat (Mac CPU)  | 2654 ms | 0.4 |
| `demos/smollm2_pleasant`     | SmolLM2-135M     | native Mat (gx10 CPU) | 876 ms  | 1.1 |
| `demos/smollm2_kv`           | SmolLM2-135M     | FFI KV-cache (Mac CPU) |  13.0 ms | **77**  |
| `demos/smollm2_kv_cuda`      | SmolLM2-135M     | FFI KV-cache (gx10 CUDA) |  11.2 ms | **89**  |
| `demos/distilgpt2_demo_kv_cuda` | gpt2-small (124M) | FFI KV-cache (gx10 CUDA) |   8.9 ms | **112** |

## Reading

* **The native Mat path is for clarity, not speed.** Ruby double-loop
  matmuls; ~1 tok/s.
* **FFI + KV-cache + ggml moves the needle by 60×–80×.**
* **gx10 CUDA over Mac CPU FFI is +14% on SmolLM2, +25% on
  gpt2-small.** At 135M parameters the model is small enough that
  GPU kernel-launch overhead eats most of the silicon advantage —
  the CPU's wide SIMD vector matmuls hold their own. The gap would
  widen on a larger model (TinyLlama 1.1B or Qwen2.5 0.5B); todo for
  next iteration.
* **Per-layer comparison:** GPT-2 small has 12 layers, SmolLM2-135M
  has 30. On CUDA: 8.9 / 12 = 0.74 ms/layer (gpt2) vs 11.2 / 30 =
  0.37 ms/layer (SmolLM2). SmolLM2's d_model 576 vs gpt2's 768 means
  smaller matmuls, hence faster per-layer.
* **CPU FFI on Mac is the most cost-effective backend at this
  scale.** No installation, no driver, no GPU. 77 tok/s on a
  laptop fan-noise budget.

## Build notes

* **CUDA archive restructure** (commit 357c544): `tinynn_ggml.c`
  used to be compiled twice (CPU and CUDA), each into its own .a
  with the same set of `tnn_*` symbols. Linker rejected multi-def
  on Linux. Restructured so `libtinynn_ggml.a` is the single source
  of truth for all common symbols (built once, no CUDA define), and
  `libtinynn_ggml_cuda.a` contains only the new
  `tinynn_backend_cuda.o` with one strong override of the weak
  `tnn_backend_cuda_init_internal()` stub in the common file. CUDA
  programs link both archives; the strong symbol wins at link time.
  Works on macOS clang and GNU ld.
* **prefer_cuda=1 fix** (commit 9b623e0): the CUDA file was passing
  `tnn_session_new(0)` (inherited from the sed-mirror of the CPU
  file). Without the post-restructure layout this had silently fallen
  through to CPU anyway because the multi-archive multi-def hadn't
  let the CUDA dispatch path actually run. Argument now matters.

## Side issues

* **CPU FFI build on gx10 still fails** with
  `undefined reference to GOMP_parallel`. Old `vendor/ggml/build/`
  built before `-DGGML_OPENMP=OFF` landed in `setup-ggml`. Fix:
  `rm -rf vendor/ggml/build && make setup-ggml`. Doesn't affect CUDA.

## Next perf steps

1. **Test on a bigger model** — TinyLlama-1.1B or Qwen2.5-0.5B. SmolLM2
   is too small for the CUDA/CPU gap to show through.
2. **Quantization** (Q8_0 / Q4_K) — would let the bigger models stay
   in RAM at f32 → q8_0 ratio. Existing `tnn_gguf_read_f32_to_doubles`
   already dequantizes; just need a converter pass.
3. **CUDA prefetch / async upload** — current `upload_weights_in 700+ ms`
   step is host→device transfer over PCIe (or NVLink-C2C on GB10).
   Not a per-token cost but it adds latency.
